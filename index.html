1. 왜 이 기술들이 필요한가

전체 그림 한 번에 보기

이 프로젝트의 목표는 접근제어 ECU에서 SecOC(보안 통신)가 잘 동작하는지 외부 진단 장비가 확인할 수 있게 하는 것입니다.

핵심 질문
"진단 장비가 ECU에게 '지금 MAC Key 제대로 들어있어? FV 정상이야?' 를 어떻게 물어보고 답을 받는가?"

이 질문에 답하는 프로토콜 체계가 바로 UDS + CanTp + AUTOSAR 스택입니다.
전체 시스템 구성
진단 장비 (Tester)
PC / JTAG
⟷
CAN 버스
물리 통신선
⟷
ECU (S32K314)
CanDrv → CanIf → CanTp → PduR → Dcm
⟷
SecOC 모듈
MAC Key / FV 관리
왜 이렇게 복잡한가
CAN 버스는 최대 64 byte만 한 번에 보낼 수 있습니다. 그런데 UDS 진단 응답은 수백 byte일 수 있어요. 그래서 CanTp가 쪼개서 보내고 받는 쪽에서 다시 붙이는 역할을 합니다. 그 위에 AUTOSAR 레이어 구조가 있고, 맨 꼭대기에서 실제 서비스(데이터 읽기/쓰기/루틴 실행)를 Dcm이 처리합니다.

2. CanTp — 큰 메시지를 잘라서 보내는 우체국

우체국 비유로 이해하기

📦 택배 없이 직접 보내기 (Classic CAN)

최대 8 byte짜리 쪽지만 보낼 수 있음.
100 byte짜리 편지는 보낼 수 없음.

🚚 택배 시스템 (CanTp)

큰 편지를 여러 조각으로 나눠서 보냄.
받는 쪽에서 순서대로 다시 붙임.
중간에 "준비됐어?" 확인도 함.

프레임 타입 4가지

타입약자역할언제 쓰나
SF Single Frame0x0N전부 한 프레임에7 byte 이하 메시지
FF First Frame0x1N첫 조각 + 전체 길이 알림멀티 프레임 시작
CF Consecutive Frame0x2N나머지 조각들FF 이후 반복
FC Flow Control0x3N"계속 보내도 돼" 응답FF 받은 후 수신 측이 보냄
N 값이 뭔가요?
SF: 0x0N 에서 N은 데이터 길이입니다. 예: 0x03 = 3 byte 데이터
CF: 0x2N 에서 N은 순서 번호입니다. 1→2→3→...→F→1→2→... 반복

멀티 프레임 실제 흐름

예: Tester가 ECU에게 35 byte짜리 Key 데이터를 보내는 경우

Tester → ECU  : FF  [전체 35 byte예요, 첫 6 byte는 이거]
ECU   → Tester: FC  [OK, 계속 보내도 돼 (ContinueToSend)]
Tester → ECU  : CF #1 [다음 7 byte]
Tester → ECU  : CF #2 [다음 7 byte]
Tester → ECU  : CF #3 [다음 7 byte]
Tester → ECU  : CF #4 [마지막 8 byte]
                ↑ 6+7+7+7+8 = 35 byte 완성!
ECU 내부: 조각들을 순서대로 붙여서 35 byte 완성 → Dcm으로 전달
FC (Flow Control) 세부 설명
FC 프레임 안에는 두 가지 중요한 값이 있습니다:
  • BlockSize (BS): 몇 개 보내고 나서 FC 기다릴지. 0이면 끝까지 쭉 보내도 됨.
  • STmin: CF 사이 최소 대기 시간 (밀리초). 0이면 최대한 빨리.

3. UDS — 진단 서비스 프로토콜

클라이언트-서버 구조

한 줄 요약
Tester(클라이언트)가 요청을 보내면, ECU(서버)가 처리하고 결과를 돌려준다.
마치 식당에서 손님이 주문하면 주방에서 요리해서 내오는 것처럼.

📤 요청 (Request)

Tester → ECU
[SID] [DID/RID] [Data...]
예: 0x22 0x12 0x34
"DID 0x1234 데이터 읽어줘"

📥 응답 (Response)

ECU → Tester
[SID+0x40] [DID/RID] [Data...]
예: 0x62 0x12 0x34 [데이터]
"SID 0x22 + 0x40 = 0x62로 응답"

SID 서비스들

SID서비스명비유본 프로젝트 용도
0x22 Read Data By Identifier 📖 책 펼쳐서 특정 페이지 읽기 ECU 상태 데이터 조회 (MAC Key 상태, FV 값 등)
0x2E Write Data By Identifier ✏️ 특정 페이지에 내용 쓰기 ECU 설정값 기록
0x31 Routine Control 🔧 버튼 눌러서 기계 동작시키기 Key Update 루틴 실행
0x14 Clear DTC 🗑️ 에러 로그 지우기 저장된 DTC 삭제
0x19 Read DTC Information 📋 에러 로그 읽기 DTC 상태 조회

DID와 RID 차이

📊 DID (Data Identifier)

데이터를 읽거나 쓴다
"이 번지수(DID)에 있는 데이터가 뭐야?"
"이 번지수에 이 값 저장해줘"

예: DID 0x1234 → "현재 FV 카운터 값"
2 byte 숫자로 구분

⚙️ RID (Routine Identifier)

동작(루틴)을 실행한다
"이 번호(RID)에 해당하는 작업을 실행해줘"
단순 데이터가 아니라 ECU 내부 로직 실행

예: RID 0x0203 → "Key Update 루틴 돌려"
2 byte 숫자로 구분

0x31 Sub Function
Routine Control(0x31)에는 Sub Function이 있습니다:
  • 0x01 = Start Routine (루틴 시작)
  • 0x02 = Stop Routine (루틴 중단)
  • 0x03 = Request Routine Result (결과 요청)

NRC — 부정 응답 코드

ECU가 요청을 처리 못하면 [0x7F] [원래SID] [NRC] 형식으로 응답합니다.

NRC이름의미
0x11serviceNotSupported해당 SID 기능 없음
0x12subFunctionNotSupported해당 Sub Function 없음
0x13incorrectMessageLength메시지 길이가 틀림
0x22conditionsNotCorrect지금 상태에서 실행 불가
0x31requestOutOfRangeDID/RID 범위 벗어남
0x78responsePending처리 중, 잠깐 기다려

4. AUTOSAR 모듈 스택 — 어떤 모듈이 무슨 일을 하나

레이어 구조

AUTOSAR CAN 스택 레이어 (위 = 애플리케이션, 아래 = 하드웨어)
Dcm — UDS 서비스 처리 (SID 파싱, DID/RID 처리)
↑↓ PduR_DcmTransmit / Dcm_StartOfReception 등
PduR — 라우터 (어디서 왔으니 어디로 보내라)
↑↓ PduR_CanTpStartOfReception / PduR_CanTpCopyRxData 등
CanTp — 멀티 프레임 조립/분해 (ISO 15765-2)
↑↓ CanTp_RxIndication / CanIf_Transmit
CanIf — CAN 인터페이스 (PDU 라우팅)
↑↓ CanIf_RxIndication / CanDrv_Write
CanDrv — 하드웨어 드라이버 (FLEXCAN 직접 제어)
↑↓ 실제 CAN 프레임
하드웨어 (S32K314 FLEXCAN 컨트롤러)
왜 이렇게 레이어를 나누나요?
각 레이어가 자기 역할만 알면 됩니다. CanDrv는 "하드웨어에서 프레임 꺼내서 위로 전달"만 하면 되고, CanTp는 "조각 맞추기"만 하면 됩니다. Dcm은 하드웨어가 뭔지 전혀 몰라도 됩니다. 교체와 유지보수가 쉬워지는 구조입니다.

RX 흐름 (Tester → ECU) 상세

Tester가 [0x22, DID=0x1234] 3 byte 요청을 Single Frame으로 보내는 경우:

  • 1
    CanDrv가 하드웨어에서 CAN 프레임 수신
    FLEXCAN 인터럽트 발생 → CanDrv가 프레임 읽음
    → CanIf_RxIndication(Mailbox, PduInfoPtr) 호출
  • 2
    CanIf가 PDU ID 기반으로 CanTp로 라우팅
    "이 PDU는 UDS용이니까 CanTp로 보내자"
    → CanTp_RxIndication(RxPduId, PduInfoPtr) 호출
  • 3
    CanTp가 PCI 필드 읽어서 SF 확인
    "첫 바이트가 0x03이니까 Single Frame, 3 byte짜리"
    → PduR_CanTpStartOfReception(id, info, 3, &bufSize) 호출
  • 4
    PduR이 Dcm으로 라우팅
    → Dcm_StartOfReception(id, info, 3, &bufSize) 호출
    Dcm: "3 byte 받을 공간 만들었어, bufSize=3 돌려줄게" → BUFREQ_OK
  • 5
    CanTp가 데이터 복사 요청
    → PduR_CanTpCopyRxData(id, info, &bufSize)
    → Dcm_CopyRxData() → "3 byte 복사 완료" → BUFREQ_OK
  • 6
    CanTp가 수신 완료 통보
    → PduR_CanTpRxIndication(id, E_OK)
    → Dcm_TpRxIndication(id, E_OK)
  • 7
    Dcm이 서비스 실행
    "SID가 0x22네, DID 0x1234 데이터 찾아서 응답 준비"
    → TX 흐름 시작

TX 흐름 (ECU → Tester) 상세

Dcm이 100 byte 응답 데이터를 Multi-Frame으로 보내는 경우:

  • 1
    Dcm이 송신 시작 요청
    → PduR_DcmTransmit(TxPduId, {SduLength=100})
    → CanTp_Transmit(CanTpTxPduId, {SduLength=100}) — E_OK 반환
  • 2
    CanTp가 FF 구성 (첫 6 byte 필요)
    → PduR_CanTpCopyTxData(id, {6 byte 요청}, NULL, &available)
    → Dcm_CopyTxData() → "처음 6 byte 줄게, 남은 거 94 byte" → BUFREQ_OK
  • 3
    CanTp가 FF 전송
    → CanIf_Transmit() → CanDrv_Write() → Tester로 FF 전송
  • 4
    Tester가 FC(CTS) 응답
    CanDrv → CanIf → CanTp: FC 수신, "계속 보내도 돼"
  • 5
    CF 반복 전송 (7 byte씩)
    매 CF마다: CopyTxData(7 byte) → CanIf_Transmit() → CanDrv_Write()
    available: 94→87→80→...→0 (0이 되면 마지막 CF)
  • 6
    송신 완료 확인 체인
    CanDrv → CanIf_TxConfirmation() → CanTp_TxConfirmation()
    → PduR_CanTpTxConfirmation(id, E_OK) → Dcm_TpTxConfirmation(id, E_OK)
    Dcm: "TX 버퍼 해제, IDLE 복귀"

PduR의 역할 — 단순 라우터

PduR이 하는 일
PduR은 "중계 창구"입니다. 자기 로직이 없고 그냥 위아래를 연결해줍니다.

CanTp가 PduR_CanTpCopyRxData()를 부르면, PduR은 그냥 Dcm_CopyRxData()를 부릅니다.
Dcm이 PduR_DcmTransmit()을 부르면, PduR은 그냥 CanTp_Transmit()을 부릅니다.

왜 이런 구조? CanTp가 Dcm을 직접 알면 CanTp ↔ Dcm 결합도가 높아집니다. PduR이 사이에서 분리해주면 나중에 한 쪽을 교체해도 다른 쪽에 영향이 없습니다.

5. DTC — 이상 진단 코드 완전 이해

DTC가 뭔가

핵심 비유
자동차 계기판의 "엔진 경고등"이 켜지면 정비사가 OBD 단말기를 꽂아서 "P0301 — 1번 실린더 실화" 같은 코드를 읽습니다.

이게 DTC입니다. ECU가 스스로 이상을 감지하면 그 원인을 코드로 메모리에 기록해두고, 나중에 진단 장비가 읽어갑니다.

DTC는 4 byte로 구성됩니다:

[Byte1][Byte2][Byte3]  = DTC 식별자 (3 byte) — 어떤 오류인지
[Byte4]                = 상태 바이트 (1 byte) — 현재 어떤 상태인지

DTC 상태 3가지

이 프로젝트에서는 bit0(testFailed)과 bit3(confirmedDTC) 두 비트만 관리합니다.

조건이 해소되어도 bit3는 남아있어서 "과거에 이런 오류가 있었다"는 기록이 유지됩니다.
진단 장비가 나중에 연결해서 볼 수 있도록 남겨두는 것입니다.

SecOC와 DTC의 연관관계

SecOC 정상 동작을 위한 전제 조건 → DTC 감시 대상
① MAC Key가 초기값이면 안 됨
공장에서 출하될 때 MAC Key는 기본값(초기값)으로 되어있습니다. 실제 운용 전에 올바른 키로 업데이트해야 SecOC가 의미있는 MAC 연산을 할 수 있습니다. 초기값 그대로면 → Initial Key DTC 발생
② Key Update가 성공해야 함
키 업데이트 루틴을 실행했는데 실패하면 → Key Update Failure DTC 발생
업데이트 성공하면 DTC Past 전환
③ Freshness Value가 한계에 가까워지면 알려야 함
FV는 재전송 공격을 막는 카운터입니다. 이 값이 최대값에 가까워지면 더 이상 새 메시지를 구별할 수 없게 됩니다. 임계값 초과 시 → FV DTC 발생

DTC 3가지 상세

① Initial Key DTC

항목내용
한 줄 요약"MAC Key/Master Key가 아직 공장 초기값이에요"
검사 시점ECU 켜질 때, Clear DTC 받은 직후, Key Update 실행 직후
발생 조건MAC Key 또는 Master Key == 초기값(공장 출하 기본값)
출하 상태Current — 키가 업데이트되기 전까지는 항상 Current
해소 조건Key Update 성공 → Past / Clear DTC → Not record

② Key Update Failure DTC

항목내용
한 줄 요약"Key Update 루틴 실행했는데 실패했어요"
검사 시점SID 0x31 Key Update 루틴 실행 직후
발생 조건루틴 결과 = 실패
출하 상태Not record — 처음엔 실패한 적 없으니까
해소 조건Key Update 성공 → Past / Clear DTC → Not record

③ Freshness Value DTC

항목내용
한 줄 요약"재전송 공격 방지 카운터가 한계에 가까워요"
검사 시점CAN-FD 통신 중 FV 갱신될 때마다 상시 모니터링
발생 조건FV 카운터 ≥ 설정된 임계값
출하 상태Not record — 처음엔 카운터가 0이니까
해소 조건FV가 임계값 아래로 내려가면 → Past / Clear DTC → Not record

6. SecOC 핵심 개념 — MAC Key, Master Key, FV

MAC Key (Message Authentication Code Key)

비유로 이해하기
MAC Key = 암호 도장

두 사람 A(ECU)와 B(다른 ECU)가 서로 같은 도장을 가지고 있습니다.
A가 메시지를 보낼 때 도장을 찍어서 보냅니다.
B는 자기 도장이랑 같은지 확인합니다. 같으면 → "진짜 A가 보낸 거" (인증 성공)

이 도장이 바로 MAC Key입니다. 대칭키(같은 키를 양쪽이 가짐)입니다. 
메시지 송신 측 (ECU A):
  [실제 데이터] + [Freshness Value] + [MAC Key]
         ↓ HMAC 연산 (해시 함수)
  [실제 데이터] + [FV 하위 비트] + [MAC 값(tMAC)]
         ↓ CAN 프레임으로 전송

메시지 수신 측 (ECU B):
  수신된 [실제 데이터] + [FV] + [자신의 MAC Key]
         ↓ HMAC 연산
  계산된 MAC == 수신된 tMAC ?
         YES → 인증 성공 (진짜 A가 보낸 것)
         NO  → 인증 실패 (위조 또는 변조)

Master Key (설계서에서 제거된 개념)

왜 설계서에서 빠졌나
Master Key는 르노 그룹 내부 키 관리 인프라 개념입니다.

원래 개념: 하나의 Master Key로 여러 채널의 MAC Key를 암호화해서 관리하는 계층적 키 구조.
하지만 이것은 르노 OEM 특화 스펙에서 나온 개념이라, 보편적인 UDS/SecOC 설계서에는 포함하지 않습니다.
본 설계서에서는 단순히 "MAC Key" (메시지 인증에 쓰는 대칭키) 만 언급합니다.

Freshness Value (FV) — 재전송 공격 방지

왜 필요한가
MAC 인증만으로는 재전송 공격(Replay Attack)을 막을 수 없습니다.

예: 공격자가 "문 열어" 명령 프레임을 녹음해뒀다가 나중에 다시 보내면?
MAC은 올바른 MAC이 붙어있으니 인증 성공 → 문이 열림 → 보안 뚫림!

해결책 = Freshness Value: 매 메시지마다 카운터(FV)를 올려서 함께 보냅니다.
수신 측은 "내가 마지막으로 받은 FV보다 커야 유효한 메시지"라고 판단합니다.
녹음된 옛날 메시지는 FV가 작으니 거부됩니다. 
시간 흐름:

메시지 1: [데이터] [FV=001] [MAC] → 수신 성공 (001 > 저장된 000)
메시지 2: [데이터] [FV=002] [MAC] → 수신 성공 (002 > 저장된 001)
메시지 3: [데이터] [FV=003] [MAC] → 수신 성공
...

공격자가 메시지 1을 다시 보냄:
[데이터] [FV=001] [MAC] → 거부! (001 ≤ 저장된 003)
FV가 임계값에 가까워지면?
FV는 결국 최대값에 도달합니다. 최대값을 넘으면 오버플로우되어 0으로 돌아옵니다.
이 시점에 공격자가 녹음해뒀던 옛날 메시지가 다시 유효해질 수 있습니다.
그래서 임계값에 가까워지면 FV DTC를 발생시켜서 관리자에게 알리는 것입니다.

용어쉬운 설명
N-SDUCanTp가 처리하는 "메시지 단위". 여러 N-PDU로 분할되기 전의 원본 메시지.
N-PDU실제 CAN 프레임 하나에 담기는 CanTp 페이로드 단위. SF/FF/CF/FC 각각이 N-PDU.
L-PDUCAN 프레임 자체. CanDrv/CanIf 레벨에서 다루는 단위.
HTHHardware Transmit Handle. 송신용 하드웨어 버퍼 번호. CanDrv_Write()에 넘겨주는 값.
HRHHardware Receive Handle. 수신용 하드웨어 버퍼 번호.
PCIProtocol Control Information. CanTp 프레임의 첫 번째 바이트. SF/FF/CF/FC 구분.
tMACTruncated MAC. MAC 값 전체가 아닌 앞 N 비트만 잘라서 CAN 프레임에 실음 (프레임 크기 제한 때문).
LSB_FVFV의 하위 비트. 전체 FV를 다 보내기엔 크니까 하위 일부만 잘라서 보냄.
availableDataPtrTX에서 "아직 안 보낸 데이터가 이만큼 남았어"를 알려주는 포인터. 0이 되면 마지막 CF.
bufferSizePtrRX에서 "이만큼 더 받을 수 있어"를 알려주는 포인터. CanTp가 FC 만들 때 씀.
SWSSoftware Specification. AUTOSAR 문서 번호 접두사. [SWS_CanTp_00298] 같은 형식.

CbCallback. CanDrv.Cb.1 처럼 쓰면 "CanDrv가 호출하는 콜백 함수"를 의미.
